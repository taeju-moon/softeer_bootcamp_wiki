# Spark Standalone 클러스터 & PySpark 병렬 처리 실습

이번 과제는 Docker Compose로 Spark Standalone 클러스터(Master 1대 + Worker 2대)를 직접 구축하고, `spark-submit`을 통해 몬테카를로 방식의 원주율(π) 추정 PySpark 잡을 제출하여 분산 병렬 처리의 흐름을 이해하는 실습 과제이다.

## 구성

- `Dockerfile` : Java + Python + Spark 3.5.1을 설치한 베이스 이미지 (Master/Worker 공용)
- `docker-compose.yml` : spark-master, spark-worker-1, spark-worker-2 3개 컨테이너 정의 (워커별 2 core / 2g 메모리)
- `jobs/pi.py` : `sc.parallelize` + `map` + `reduce`로 원 안에 떨어진 점의 비율을 계산해 π를 추정하는 PySpark 잡
- `submit.sh` : 실행 중인 `spark-master` 컨테이너에 `spark-submit`을 대신 실행해주는 래퍼 스크립트
- `data/output/pi_result` : 잡 실행 결과가 저장된 CSV (`partitions,samples,hits,pi_estimate`)

## 이번 프로젝트를 하며 배운 점들

1. **Standalone 클러스터에서 Master와 Worker의 역할 분리** 하나의 Docker 이미지로 `start-master.sh`, `start-worker.sh` 커맨드만 다르게 주어 Master와 Worker 역할을 나눌 수 있다는 점을 배웠다. Worker는 기동 시 `spark://spark-master:7077`로 접속해 스스로 등록하고, Master 로그에서 `Registering worker ...` 메시지로 두 워커(각 2 core)가 정상적으로 클러스터에 합류했음을 확인할 수 있었다.

2. **`sc.parallelize`의 파티션 수가 곧 병렬 처리 단위** `pi.py`에서 전체 샘플(점)을 `partitions` 인자만큼 나눠 `sc.parallelize(range(1, n+1), partitions)`로 분할하면, Spark가 이를 여러 executor에 태스크로 흩뿌려 동시에 계산한다는 것을 이해했다. 파티션 수를 늘릴수록 더 잘게 쪼개져 워커들에 고르게 분배되는 것을 Master 로그의 `Launching executor` 메시지로 확인할 수 있었다.

3. **드라이버(Client 모드)와 실행 결과의 분리** `spark-submit --deploy-mode client`로 제출하면 드라이버는 `spark-master` 컨테이너 안에서 실행되고, 실제 연산(map/reduce)은 Worker의 executor들이 나눠 수행한다는 점을 배웠다. 계산 결과(π 추정치, 샘플 수, 히트 수)는 `result_df.coalesce(1).write.csv(...)`로 한 개의 CSV 파일로 합쳐져 저장되는데, 이를 통해 분산 처리된 결과라도 최종적으로는 하나의 output으로 모을 수 있음을 확인했다. (예: `partitions=10`, `samples=1000000`일 때 `pi_estimate ≈ 3.14412`)

4. **로컬 스크립트 반복 실행과 클러스터 배치 실행의 차이 체감** 단일 프로세스에서 순차적으로 점을 찍어 π를 계산하는 것과 달리, 클러스터에 잡을 "제출(submit)"하고 Web UI(`localhost:8080`)에서 앱 등록 → executor 실행 → 완료까지의 흐름을 지켜보면서, 분산 병렬 처리는 단순히 코드를 여러 번 돌리는 것이 아니라 클러스터 매니저가 자원을 스케줄링해주는 별도의 실행 모델이라는 것을 체감했다.
